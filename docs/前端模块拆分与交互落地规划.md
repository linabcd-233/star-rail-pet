# 前端模块拆分与交互落地规划

本文档在《交互与表情仲裁设计》的产品/技术规则之上，规划 **`pet-web/src`** 的目标目录、各文件职责，以及**先拆什么、后接什么**的顺序。便于评审与分步实施。

**关联文档**：`docs/交互与表情仲裁设计.md`

---

## 1. 目标原则

1. **先拆现有、再上新交互（强烈推荐）**  
   - **阶段 A**只做「搬家」：行为与现网一致，**不**增加过山车、星星眼、仲裁器等新逻辑。  
   - 每完成一小步就做**回归验证**（浏览器 + Tauri：`穿透 / 拖窗 / 番茄钟 / 气泡 / 右键菜单 / 轮播`）。  
   - **阶段 B**再在已稳定结构上实现《交互与表情仲裁设计》。  
   - 这样出问题时可明确：是**拆分回归**还是**新功能**，避免双线并行难以归因。

2. **bootstrap 极薄**：入口只负责挂载 DOM、创建 `SpineCanvas`、把回调接到各模块。

3. **交互决策与 Spine 生命周期分离**（阶段 B）：仲裁、输入信号尽量**不依赖** `SpineCanvas` 细节，只依赖 `animState` / 动画名等窄接口。

4. **Tauri 与浏览器分支集中**：穿透、光标轮询等放在单独模块。

5. **渐进迁移**：阶段 A 允许 `main.ts` 仍略大；**阶段 B 起**新交互代码**禁止**再堆回单文件巨石。

## 2. 建议目录结构（`pet-web/src`）

```
pet-web/src/
├── main.ts                    # 入口：import 样式、组装、启动 SpineCanvas
├── style.css                  # 全局样式（保持或后续按组件拆，非本阶段重点）
├── app/
│   ├── dom.ts                 # root.innerHTML 或模板字符串、querySelector 集中处（可选：迁到 HTML 模板）
│   ├── refs.ts                # 各 DOM 引用类型与获取（pomoPanel、canvas、speechBubble…）
│   └── index.ts               # 可选：统一 export app 初始化函数 initApp()
├── spine/
│   ├── assets.ts              # SKEL、ATLAS、动画名常量（IDLE、TALK、EMOJI_*）
│   ├── camera.ts              # stableBounds、fitCamera、captureStableCameraFromSkeleton
│   ├── hitTest.ts             # clientPointToCanvasPixels、hitTestAtClientPoint（依赖 gl + canvas + skeleton）
│   └── canvasApp.ts           # SpineCanvas 配置对象：load / init / update / render / error（从 main 抽离）
├── tauri/
│   ├── env.ts                 # isTauri 判定
│   ├── windowChrome.ts        # setSize、scale、LogicalSize、scheduleWindowResizeToScale
│   ├── pointerPassthrough.ts  # setIgnoreCursorEventsSafe、enable/disableBackgroundClickThrough
│   └── pointerLoop.ts         # tauriPointerTick、ensureTauriPointerLoop、与 cursorPosition 轮询
├── interaction/               # 【阶段 B 再建】阶段 A 不要创建，避免拆的同时改行为
│   ├── types.ts               # AnimMode、InputSnapshot、ArbiterConfig 等
│   ├── animArbiter.ts         # 优先级、talk/sweat/star/idle、*Until、冷却
│   ├── signals.ts             # vy、inside、dragging、dt → sweatUntil、wiggleEnergy
│   ├── windowMotion.ts        # Tauri：outerPosition、vy / EMA
│   └── applyAnimation.ts      # 可选：统一 setAnimation 出口
├── pomodoro/
│   ├── state.ts               # pomoPhase、remaining、defaultDuration、parse/format
│   ├── ui.ts                  # updatePomoTimeDisplay、syncPomoPanelWidthFromButtons、按钮状态
│   ├── panelDrag.ts           # setupPomoPanelDrag、clampPomoPanelPosition
│   └── speechBridge.ts        # 与 beginSpeech 的衔接（番茄事件 → 仲裁 talk 请求）
├── ui/
│   ├── speechBubble.ts        # show/hide、layoutSpeechBubble（依赖 bone + camera + root 几何）
│   ├── contextMenu.ts         # 角色右键菜单、命中与穿透配合
│   └── gaze.ts                # gaze 状态、syncGazeFromClient、update 里骨骼旋转（可选与 spine 合并）
└── utils/
    └── math.ts                # clamp、lerpExp 等无业务工具函数
```

**说明**：

- **`app/`**：解决「模板 + 一堆全局 querySelector」的散乱；若短期不想动模板，可只建 `refs.ts`，`innerHTML` 暂留 `main.ts`。
- **`spine/hitTest.ts`**：与 `spine/canvasApp.ts` 有耦合（需 `sp`、`skeleton`、`canvas`），可二合一为 `spine/runtime.ts`，上表为逻辑拆分示意。
- **`interaction/`**：**仅阶段 B 创建**；阶段 A 拆分时不出现该目录。新交互（过山车、星星眼、仲裁）**全部写在这里**。
- **`pomodoro/`**：番茄钟与 DOM 强相关，独立后 `main.ts` 明显变薄。
- **`tauri/`**：所有 `@tauri-apps/api/window` 的副作用集中，便于非 Tauri 构建时 tree-shake 或 mock。

## 4. 实施顺序：阶段 A（仅拆分）→ 阶段 B（新交互）

### 阶段 A — 只迁移、不加功能

**原则**：每一子阶段结束都打勾验证（见下文「阶段 A 回归清单」），再进入下一子阶段。

| 子阶段 | 内容 | 说明 |
|--------|------|------|
| **A0** | `utils/math.ts`：`clamp`、`lerpExp` | 纯剪切，无行为变化 |
| **A1** | `spine/assets.ts`：`SKEL`、`ATLAS`、`IDLE`/`TALK`、台词数组等常量 | 无行为变化 |
| **A2** | `spine/camera.ts`：`stableMid*`、`captureStableCameraFromSkeleton`、`fitCamera` | 通过传入 `canvas`/`skeleton`/`sp` 避免隐式全局 |
| **A3** | `spine/hitTest.ts`：`clientPointToCanvasPixels`、`hitTestAtClientPoint` | 签名可接收 `getSkeleton()` 闭包，逻辑一字不改 |
| **A4** | `tauri/env.ts`、`pointerPassthrough.ts`、`windowChrome.ts`、`pointerLoop.ts` | 从 `main` 剪切；`isTauri` 守卫保持 |
| **A5** | `ui/speechBubble.ts`、`ui/contextMenu.ts`（+ 可选 `ui/gaze.ts`） | `layoutSpeechBubble`、`beginSpeech` 相关 DOM 与事件 |
| **A6** | `pomodoro/*`：状态、UI、拖面板、`updatePomoTimeDisplay` 等 | 仍由 `canvasApp` 的 init 里注册回调，**不改番茄逻辑** |
| **A7** | `spine/canvasApp.ts`：`SpineCanvas` 的 `load/init/update/render/error` 大对象 | `main.ts` 仅 `import { createCanvasApp } …` + `new SpineCanvas(canvas, app)` |
| **A8** | `app/refs.ts`（及可选 `dom.ts`） | 集中 `querySelector`，减少 `main` 散乱 |

**阶段 A 结束前**：`interaction/` **不存在**；`main.ts` 尽量压到「胶水层」目标行数（如 &lt; 250 行），**动画切换路径与拆分前完全一致**（仍为 `beginSpeech`、`cycleMode`、`update` 内现有 `setAnimation`）。

### 阶段 A 回归清单（建议逐项手测）

- [ ] 浏览器 `npm run dev`：加载、动画、`?anim` / `?cycle`
- [ ] Tauri：`npm run tauri:dev`
- [ ] 像素穿透：透明区穿透、本体与番茄钟/右键菜单不穿透
- [ ] 左键拖窗（本体命中）
- [ ] 番茄钟：开始/暂停/继续/重置、台词与气泡、结束回 idle
- [ ] 气泡跟随头部、与番茄钟避让
- [ ] 右键菜单：开关计时器、点击与穿透
- [ ] Ctrl+滚轮缩放（若已启用）

### 阶段 B — 在稳定结构上实现《交互与表情仲裁设计》

1. **新建** `interaction/` 目录：`types.ts`、`animArbiter.ts`、`signals.ts`…  
2. 把 **现有** `setAnimation` 调用逐步收口到仲裁器（或 `applyAnimation.ts`），仍**只**表现当前已有行为（idle / talk / cycle emoji），确认无回归。  
3. 再接入：`windowMotion`（`vy`）、星星眼能量、流汗 TTL 等**新规则**。

阶段 B 若出 bug，可对比阶段 A 的 tag/commit，快速判断是否为仲裁/新信号问题。

## 5. 依赖关系（简图）

**阶段 A 结束时**（无 `interaction/`）：

```
main.ts
  → spine/canvasApp.ts → spine/camera.ts, spine/hitTest.ts, spine/assets.ts
  → tauri/*
  → pomodoro/* , ui/*
```

**阶段 B 扩展后**：

```
main.ts
  → spine/canvasApp.ts → …
  → interaction/animArbiter.ts ← interaction/signals.ts ← InputSnapshot
  → interaction/windowMotion.ts（Tauri）→ tauri/pointerLoop.ts
```

**注意**：`hitTest` 需要运行时 `skeleton` + `gl`，仲裁器**不应**直接 import `hitTest`；由外层把 `inside: boolean` 传入 `InputSnapshot`。

## 6. 风险与规避

| 风险 | 规避 |
|------|------|
| 循环依赖（arbiter ↔ spine） | 仲裁只依赖「动画名字符串 + 时间戳」，Spine 侧只调用 `apply` |
| Tauri 与 Vite 浏览器双构建 | `tauri/*` 内对 `getCurrentWindow` 做 `isTauri` 守卫或动态 import |
| 迁文件时行为漂移 | 每阶段迁一块即跑一次 `tauri dev` / `npm run dev` |
| 拆分与上新功能同时进行 | **禁止**：按阶段 A → 阶段 B，并用清单验收 |
