这种spine到底是怎么做的呢？银枝小人出现在官方前瞻视频和周年网页小动画里，也有过其他动作，最初素材获取难道要抠图？然后呢，怎么拆分的骨骼怎么弄的啊

下面用「行业里常见做法」说明，不针对米哈游内部真实流程（那属于未公开管线），但和你在前瞻、网页里看到的那类 Spine/Q 版小人 通常是怎么来的。

1. 官方素材一般不是从视频里抠出来的
正规做法是美术按设定 直接画分层源文件（Photoshop / Clip Studio 等）：

身体、四肢、头发、衣服、五官等 各一层或几块，方便后面单独绑骨、做网格。
前瞻视频、网页小动画用的，多半是 同一套或衍生 PSD → 进 Spine（或同类工具）→ 导出给程序/视频包装。
从已压缩的视频里抠图再还原 Spine，画质差、缺背面和遮挡关系，官方不会当主流程；民间二创有时会从截图描改，那是另一回事。

2. 「拆分」到底拆什么？
不是神秘黑科技，就是把一张角色拆成多块可动的图：

概念	做什么
切片 / 部件
头、上臂、前臂、手、躯干、腿、发片、披风等，每块一张小图（或一张大图里多 region）。
网格（Mesh）
矩形图可以切成多边形网格，顶点绑在多根骨上，转弯、呼吸时 形变更自然（你工程里 mesh 就是这类）。
骨骼
在 Spine 里建父子链：例如 root → 身体 → 胸 → 颈 → 头 → 头发/五官，四肢再各一条链。
权重
网格顶点跟哪根骨、多大比例跟，决定弯关节时皮怎么皱。
所以：先有分层美术 → 再搭骨、绑权重 → 最后 K 动画。「拆」主要是 拆图层 + 必要时拆网格，不是从视频里拆。

3. 动画（站、走、表情）怎么来的？
同一套骨骼上可以挂 很多动画剪辑：idle、walk、sit 等，都是 在时间轴上 K 关键帧（旋转、位移、换附件、网格变形等）。
前瞻里的镜头可能是 Spine 导出序列帧 / 视频合成，或 引擎里实时播；网页小动画可能是 Spine 运行时、Lottie、Canvas 等，取决于项目。
不同动作 = 同一 rig 的 不同 animation，或 不同皮肤/附件，不是「再抠一次图」。

可以，完全能靠你现在这套「骨骼 JSON + 每帧算姿态」的代码做，而且不必等新动画资源。下面是一份可直接当开发说明用的设计规划（先架构，再分阶段落地）。

1. 可行性结论
能力	做法
待机
继续播现有 idel（apply_animation_to_bones）
看向光标 / 被点晃 / 假走 / 假起立
在同一帧里，在 Spine 算好的骨骼上再叠加一层「程序位移」（改 x/y/rotation 等）
你已有：apply_animation_to_bones → compute_bone_worlds → transform 约束 → 画 mesh。只要在 apply_animation 之后、算世界矩阵之前插入「程序层」，就能实现。
2. 核心思路：两层合成
每帧：
    bones = apply_animation_to_bones(模板, idel, t)     # 底层：官方待机
    bones = apply_procedural_overlay(bones, 状态机, 输入, dt)  # 上层：程序增量
    worlds = compute_bone_worlds(...)
    worlds = apply_transform_constraints_translate_world(...)
    绘制
原则：程序层只动少数几根骨（root、头/颈、可选胸骨），增量要小，避免和 idel 里大动作打架；若某根骨在 idel 里动得很猛，程序层就只加小角度（例如 ±8°）做「看向」。
3. 输入与坐标
pygame：MOUSEMOTION、MOUSEBUTTONDOWN、窗口宽高。
光标：屏幕坐标 → 与角色锚点（建议用 root 世界位置投影到屏幕，或固定用窗口中心偏下的「脚底」）算向量，得到目标水平角或屏幕偏移。
注意：你当前是正交投影 + 可能 Y 轴翻转，「看向」要把 屏幕 dx/dy 转成骨骼局部期望旋转时，集中写一个小函数 screen_delta_to_head_nudge(dx, dy)，避免每处重复、符号搞反。

5. 各功能怎么落到骨骼上（实现要点）
5.1 看向光标（将就版）
选骨：颈/头链上的一根（你工程里常见是类似 左看右看、或 身体4 等，需打印骨骼列表定一根「转一点整张脸都跟」的骨）。
行为：根据鼠标相对角色的 水平方向 算目标 rotation 偏移，再限制在 ±max 度；用 smooth 跟上（例如指数平滑：cur += (target - cur) * 1 - exp(-k*dt)）。
垂直方向可只做很小修正或忽略，避免和 idel 冲突。
5.2 被点晃一下
在 MOUSEBUTTONDOWN 且命中角色粗略包围盒（可用当前 mesh 的 min/max 或固定矩形）时：
给 root 或 身体 初速度 (vx, vy)，或给头骨一个短旋转脉冲。
每帧：v *= damping，pos += v*dt，直到 |v| 很小。
叠加在 已含 idel 的 x,y 上（先采样动画，再加偏移）。
5.3 假走（walk）
逻辑位置：logical_x, logical_y（或只沿 x）向目标 lerp 或带最大速度。

## 关键概念：`AnimationState` 的“轨道”(track)

- track 0：通常放“底层常驻”的动作（比如 `idel` 呼吸）。
- track 1 / 2 ...：放“短暂覆盖/表情动作”（`emoji_0/3/4` 眨眼、`emoji_2` 流汗、`emoji_5` 说话等）。
- Spine 的应用顺序是 先应用低轨道，再应用高轨道；高轨道对它“有时间轴的那些属性”会覆盖低轨道，从而实现“叠加/覆盖”。

这就对应你说的“多个模组来回切换”。

------

## 方案 1：一边呼吸（idel）一边表情动作（叠加），并且定时切换

### 做法

1. `track 0` 永远循环播放 `idel`（呼吸）。
2. `track 1` 播放一个表情动画（比如 `emoji_2` 流汗）持续几秒，然后淡出/清空 track 1，再换下一个表情。

### 你会用到的 API（在 `main.ts` 里）

- `animState.setAnimation(0, "idel", true)`
- `animState.setAnimation(1, "emoji_2", true/false)`（看你表情要不要循环）
- `animState.setEmptyAnimation(1, mixSeconds)` 或 `animState.clearTrack(1)`（让表情退出）
- `asData.setMix("idel", "emoji_2", 0.2)` / `asData.setMix("emoji_2", "idel", 0.2)`（切换更顺滑）

### 注意点（很重要）

- 只有当 `emoji_x` 不去动全身呼吸用的同一批骨骼/slot 时，叠加才会“像你想的那样：呼吸 + 表情”。
- 如果某个 `emoji` 动了躯干骨骼，放到 track 1 就会把呼吸覆盖掉（你现在看到“表情时没有上下起伏”很可能就是因为它覆盖了相关骨骼/形变）。
- 这种情况下你仍能叠加，但需要进一步做“只让表情影响脸/嘴/汗符号相关 slot”的方案（后续可以做：限定 slot 范围或用更细的动画拆分；官方运行时支持 slot range 绘制，但“只应用动画到部分骨骼/slot”要更深入一点实现）。

### 做法 B（更“动画系统”的写法：队列）

## 要不要先拆模块？

结论：不必为「八百行」单独先做一轮大拆分；更建议「边加交互边抽出 1～2 个文件」，或至少先把仲裁器放进独立文件。

理由简述：

1. 复杂度主要来自交叉，不是行数
   当前 `main.ts` 已混在：DOM 模板、番茄钟、气泡布局、Tauri 穿透、命中测试、拖窗、 gaze、相机、轮播、`SpineCanvas` 生命周期。再塞 仲裁 + 窗体速度 + 晃动，单文件可读性会明显下降，但问题本质是职责混杂，不是「超过 800 行」这个魔法数字。
2. 先大拆再写功能 = 高风险、难 review
   一次拆成很多小文件容易引入行为差异（事件顺序、`this`、闭包里的 `skeleton`/`animState`）。若没有自动化测试，性价比一般。
3. 性价比高的折中
   - 方案 A：新建 `pet-web/src/interaction/animArbiter.ts`（或 `expressionController.ts`），只放纯状态 + 决策 + 常量，通过参数传入 `now`、`inside`、`dragging`、`vy`、`speechLocked` 等，返回「应播的动画名」或 `apply` 回调；`main.ts` 仍负责取 Tauri 数据、调 `setAnimation`。新增交互代码 主要落在该文件，`main.ts` 只多几十行胶水。
   - 方案 B：第二迭代再把 `tauriPointerTick` + hitTest + 穿透 抽到 `tauriInput.ts`，把 番茄钟 抽到 `pomodoro.ts`。
4. 什么时候值得「整文件拆分」
   当你要同时改：仲裁 + 多路输入 + UI 面板，且多人协作或要单测仲裁逻辑时，再拆成 `main.ts`（bootstrap）+ `spineApp.ts` + `pomodoro.ts` + `interaction/*` 更划算。

### 2）你说的 `maxPitchDeg` 和「另一个」——它们本来就不是一回事

这些 没有 被我合成一个数，也不需要合成，因为职责不同：

| 东西                             | 干什么                                                       |
| :------------------------------- | :----------------------------------------------------------- |
| `maxPitchDeg`                    | 俯仰角的硬上限（度）                                         |
